# Источники данных ковки

Формат и набор файлов соответствуют приложению **версии 2.1.1** (см. корневой [`CHANGELOG.md`](../CHANGELOG.md)).

Файлы в этой папке — **единственный источник правды** для материалов, секций (источников), суффиксов и точек G. После правок запустите:

```bash
npm run generate-smithing
```

(команда также выполняется автоматически перед `npm run build`.)

### Шаблон нового `G_points - <материал>.txt`

Структура строк (заголовки секций и пары `English / Русский`) копируется из **`materials_suffixes.txt`**. В строках предмета после **`=`** ничего не подставляется — только вручную дописать число G (0..150), в том же формате, что и в уже заполненных файлах.

```bash
npm run generate-g-points-template -- "Red Alloy"
```

Английское имя — **как левая часть** строки в **`materials`** (до ` / `). Имя файла: `G_points - <slug>.txt` (slug из английского имени, как у существующих материалов). Если файл уже есть, используйте **`--force`** для перезаписи или **`--stdout`**, чтобы вывести шаблон в консоль.

### Проверки в `npm run test`

Используется тот же слой слияния, что и **`npm run generate-smithing`** (`buildSmithingDatasetFromDataSource`).

**1. Валидация состава** — `src/lib/smithingDataSource.validation.test.ts`  
Падает с текстом **почему не прошла валидация**, если есть **любая** запись с `severity: error` (в т.ч. новые коды после доработки генератора):

| Код | Смысл |
|-----|--------|
| **`UNKNOWN_SECTION_HEADER`** | В `materials_suffixes.txt` или `G_points` строка `X / Y:` не совпадает с заголовком из **`sources`** (например, добавили `Plate / Пластина:` в G, но забыли строку в **`sources`**). |
| **`G_KEY_NOT_IN_SUFFIXES`** | В `G_points` указан предмет, которого нет в **`materials_suffixes.txt`** для этой секции. |
| **`G_OUT_OF_RANGE`** | Число G в строке не в диапазоне **0..150** (как в игре и в расчёте путей). |
| **`NO_MATERIALS_WITH_G`**, **`NO_ITEMS_AFTER_MERGE`** | Нет материалов с G-файлами или после слияния нет ни одного предмета. |

Строка в **суффиксах** без ни одного G в материалах **ошибкой не считается** (так и задумано: не у всех сплавов есть все предметы).

**2. Покрытие G → UI** — `src/lib/smithingDataSource.uiCoverage.test.ts`  
Считает **только строки предметов** в `G_points` (строки с числом G; **заголовки секций не считаются**). Проверяет, что каждая такая строка попала в слитый конфиг с тем же G, и что **число этих строк = сумме** всех `gByMaterial` по предметам. При расхождении тест выводит список **чего не хватает** (файл, строка, материал, секция, ключ).

**3. Совпадение с репозиторием** — `src/lib/smithingDataSource.generatedSync.test.ts`  
Сливает `data_source/` в памяти и сравнивает результат с **`src/generated/smithing-data.ts`**. Если правили только txt/манифесты и забыли `npm run generate-smithing`, тест упадёт (в репозитории останется старый сгенерированный конфиг).

Предупреждения (**`warn`**) не валят тесты: неизвестный файл `G_points`, материал в **`materials`** без G-файла и т.п.

Чтение файлов: при отсутствии каталога или файла сообщение об ошибке будет вида `data_source: не удалось прочитать «…» (путь): …`.

## Файлы манифеста

| Файл | Назначение |
|------|------------|
| **`materials`** | Материалы: английский ключ, русская подпись, путь к иконке (`public/...` или `/icons/...`). Порядок строк задаёт сортировку в UI. |
| **`sources`** | Секции ковки: `id` в коде — **`ingot`**, **`rod`**, **`double_ingot`**, **`plate`** (плюс `sortIndex`, подпись в UI, **точная** строка заголовка, как в `materials_suffixes.txt`). |
| **`materials_suffixes.txt`** | Суффиксы по предметам, с заголовками секций из `sources`. |
| **`G_points - <material>.txt`** | Точки G по материалу (`<material>` совпадает со slug из английской части в `materials`). |

Добавление **нового материала**: строка в `materials` + файл `G_points - <slug>.txt` (как у существующих).

Добавление **новой секции**: строка в `sources` (новый `id` и строка заголовка) + такой же заголовок в `materials_suffixes.txt` и в каждом `G_points - *.txt`, где нужны рецепты.

## Общие правила

1. **Основной язык в строках предметов — английский** (ключи для кода). **Русский — подпись** для интерфейса.
2. Формат пары: `English / Русский` (пробелы вокруг `/` желательны).
3. Английское имя в `materials_suffixes.txt` и в `G_points` для одного предмета должно **совпадать** (регистр для slug не важен).
4. Строка **без числа G** в файле материала означает «рецепта нет» — предмет для этой пары не попадёт в список.

### Неизвестный заголовок внутри файла

Строка вида `X / Y:` **должна** совпадать с одной из строк в **`sources`**. Иначе **`npm run test`** и **`npm run generate-smithing`** завершатся с **ошибкой** (`UNKNOWN_SECTION_HEADER`) — иначе данные окажутся в неверной секции и предметы уйдут не в тот источник в UI.

## `materials`

Одна строка — один материал:

```text
Copper / Медь = public/icons/material-copper.svg
```

Путь можно писать с обратными слэшами (Windows); префикс `public/` опционален — в приложении путь нормализуется под каталог `public/`.

## `sources`

Табуляция между колонками:

```text
ingots	10	Слиток	Ingot / Слиток:
```

Колонки: **`id`**, **`sortIndex`**, **подпись в UI**, **заголовок** (как в `materials_suffixes.txt`, после `trim` — посимвольное совпадение).

Строки с `#` в начале — комментарии.

## `materials_suffixes.txt`

### Строка предмета

Рекомендуемый формат:

```text
English Name / Русское имя = +13 -3 +2
```

Поддерживается вид без `=` (дельты через пробел после русской подписи).

## `G_points - <material>.txt`

Имя файла: `G_points - copper.txt` → материал `copper` (как в `materials` после slug).

### Строка с точкой G

```text
Pickaxe Head / Оголовье кирки = 100
```

## Выход

Генератор пишет `src/generated/smithing-data.ts`: в UI — **русские** подписи, в коде — **английские** slug (`ingots_pickaxe_head` и т.д.).
