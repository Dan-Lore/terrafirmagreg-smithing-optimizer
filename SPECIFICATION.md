# Спецификация: оптимизатор ковки (точка G)

Актуальная версия продукта в репозитории: **`2.1.0`** (см. [`CHANGELOG.md`](CHANGELOG.md) — отличия от последнего снимка на GitHub `0.1.0` / `g-point-steps-spa`).

## 1. Математическая постановка

### 1.1 Команды и граф

Фиксированный набор дельт (как в игре):

- `+2`, `+7`, `+13`, `+16`
- `-3`, `-6`, `-9`, `-15`

Состояния — целые **0..150**. Из состояния `x` допустим переход в `x + δ`, если результат снова в **0..150**.

Требуется **минимизировать число шагов** (невзвешенный кратчайший путь в этом ориентированном графе).

### 1.2 Ковка предмета

Для пары **(материал, источник, предмет)** заданы:

- целевая точка **G** (0..150);
- **суффикс** — последовательность из 1–3 команд, с которой **заканчивается** полная ковка.

Суффикс зависит **только от предмета** (и источника), не от материала. **G** зависит от **материала и предмета** (и источника).

Пусть суффикс в порядке применения — `d₁, …, dₖ`. Состояние **непосредственно перед** первой командой суффикса:

\[
s_0 = G - d_k - d_{k-1} - \cdots - d_1
\]

(обратное «снятие» команд с конца).

**Полная** последовательность: кратчайший путь от **стартового** состояния режима до `s_0`, затем суффикс. На каждом шаге значение должно оставаться в **0..150**.

### 1.3 Режимы приложения (блок «Режим ковки»)

1. **Из любой точки в точку G**  
   Задаются **старт** и **цель G** (оба **0..150**). Считается кратчайший путь между ними; предмет и суффикс не участвуют.

2. **Из точки G в предмет**  
   Старт = **G** текущего выбранного рецепта. Префикс: кратчайший путь от `G` до `s_0`, затем суффикс. Выбор **материала** в UI отключён.

3. **Из любой точки в предмет** (по умолчанию в UI)  
   Старт = введённое целое **0..150** (по умолчанию **0**). Префикс: кратчайший путь от этого значения до `s_0`, затем суффикс.

Поиск кратчайшего пути — **BFS** на том же графе (`src/lib/pathfinding.ts`).

### 1.4 Предвычисление `paths.ts`

Скрипт `scripts/precompute-paths.ts` строит BFS **от 0** и записывает `src/generated/paths.ts` — таблицу кратчайших путей в каждую вершину 0..150. Текущий UI ковки **не использует** эту таблицу для префикса (пути считаются через BFS от выбранного старта). Автотесты сверяют `paths.ts` с `shortestPath(0, g)` для всех `g`, чтобы таблица не расходилась с кодом после `npm run precompute`.

## 2. Данные: `data_source/`

Формат строк описан в **`data_source/README.md`**.

Кратко:

- В строках предметов **основной язык — английский** (идентификация и стабильные ключи), **русский** — подпись для UI.
- Шаблон: `English / Русский …`
- **Материалы** (`materials`) и **секции/источники** (`sources`) задаются отдельными манифестами; список секций и подписи в UI для фильтра «источник» не дублируются в коде генератора.
- Заголовки секций в `materials_suffixes.txt` и в `G_points - *.txt` должны **точно** совпадать (после `trim`) со строками в `sources`.
- Суффиксы: `data_source/materials_suffixes.txt`. Генератор: `scripts/generate-smithing-data.ts` → `src/generated/smithing-data.ts`.

В интерфейсе отображаются **русские** названия; **id** предметов в коде — `{sourceId}_{slug_en}`, например `ingot_pickaxe_head`, `plate_unfinished_boots` (slug из английской части; `sourceId` из `sources`: `ingot`, `rod`, `double_ingot`, `plate`).

## 3. Результат в UI

Для валидного выбора показываются:

- G, суффикс, состояние перед суффиксом;
- префикс (минимум шагов), суффикс, полная последовательность;
- число шагов префикса и всего.

При отсутствии пути или нарушении ограничений — сообщение об ошибке.

## 4. Сборка и деплой

- `npm run build` выполняет `generate-smithing` и `vite build`.
- GitHub Pages: workflow в `.github/workflows/pages.yml`, `VITE_BASE_URL` задаётся под подкаталог репозитория (`vite.config.ts`).
